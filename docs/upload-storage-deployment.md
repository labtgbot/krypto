# Public Upload Storage Deployment Guards

Krypto stores user-supplied files in public web paths. Application-side upload
validation rejects executable extensions before files are moved, but production
deployments also need web-server rules that keep these directories static-only.

## Current Upload Directories

Apply the upload execution guard to every directory below:

- `public/user`
- `public/logo`
- `public/chat`
- `public/identity`
- `public/proof`
- `public/bank-proof`

`public/qrcode` is generated by the application and is not an upload intake, but
it is still covered by the repository-level Apache guard because it lives below
`public`.

## Supported Web Servers

- Apache: supported through the committed `public/.htaccess`.
- Nginx with PHP-FPM: supported when the server block includes the Nginx guard
  below before the general PHP handler.
- IIS: supported when Request Filtering and directory browsing controls are
  enabled for the upload paths.
- Reverse Proxy before PHP-FPM: supported when the proxy applies the same deny
  rule before forwarding PHP requests upstream.

## Application Content Validation

The shared upload guard validates both the extension and the detected content
before files are moved below `public/*`.

- Extension allowlists are synchronized with the application MIME map for
  `jpg`, `jpeg`, `png`, `gif`, and `pdf`.
- MIME Sniffing uses `finfo_file()` with `mime_content_type()` as a fallback.
- Images must also decode through `getimagesize()`, match their final
  extension, have non-zero dimensions, and stay within the 12000px dimension
  ceiling.
- PDFs must sniff as PDF content, start with `%PDF-`, and include a `%%EOF`
  marker near the end of the file.

The PDF check is intentionally a minimal content validation boundary. It is not
a PDF sanitizer, JavaScript remover, malware scanner, OCR verifier, or deep
parser. Add a separate sanitizer/scanning pipeline before accepting untrusted
PDFs in deployments that require that level of assurance.

## Apache

The repository includes `public/.htaccess`. It disables directory listing,
turns off `mod_php` execution when available, and denies PHP-like extensions
under `public`.

Confirm the deployment allows `.htaccess` overrides for `public`. If Apache is
configured with `AllowOverride None`, move the equivalent rules into the vhost.

## Nginx

Place the deny rule before the generic PHP location. The static upload location
keeps normal files readable while preventing directory listing.

```nginx
location ~* ^/public/(?:user|logo|chat|identity|proof|bank-proof)/.*\.(?:php[0-9]?|phtml|phar)(?:/|$) {
    return 404;
}

location ~* ^/public/(?:user|logo|chat|identity|proof|bank-proof)(?:/|$) {
    autoindex off;
    try_files $uri =404;
}

location ~ \.php(?:/|$) {
    include fastcgi_params;
    fastcgi_pass unix:/run/php/php-fpm.sock;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}
```

If Krypto is mounted below a subpath, prepend that subpath before `/public/` in
both upload locations.

## IIS

Use Request Filtering to reject PHP-like extensions and keep directory browsing
disabled. The example can be placed in a `web.config` under each upload
directory, or translated into matching `<location path="...">` entries in the
site-level configuration.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <directoryBrowse enabled="false" />
    <security>
      <requestFiltering>
        <fileExtensions allowUnlisted="true">
          <add fileExtension=".php" allowed="false" />
          <add fileExtension=".php3" allowed="false" />
          <add fileExtension=".php4" allowed="false" />
          <add fileExtension=".php5" allowed="false" />
          <add fileExtension=".phtml" allowed="false" />
          <add fileExtension=".phar" allowed="false" />
        </fileExtensions>
      </requestFiltering>
    </security>
  </system.webServer>
</configuration>
```

Do not configure wildcard or extensionless PHP handlers for these upload
directories. If the IIS site uses a global PHP handler, verify these directory
rules still reject direct requests for `*.php`, `*.phtml`, and `*.phar`.

## Reverse Proxy / PHP-FPM

When Nginx, Apache, a CDN, or another Reverse Proxy sits in front of PHP-FPM,
apply the upload deny rule at the first component that receives public traffic.
The proxy must reject PHP-like upload paths before forwarding anything to the
upstream PHP handler.

## Allowed Upload Reads

These guards block only PHP-like extensions and directory indexes. Valid static
uploads such as `jpg`, `jpeg`, `png`, `gif`, and `pdf` must continue to be
served as files. After changing server configuration, verify:

- `GET /public/user/<id>/avatar.jpg` returns the image.
- `GET /public/identity/<id>/document.pdf` returns the document when the app
  has authorized/publicly exposed that URL.
- `GET /public/chat/<room>/payload.php` returns 403 or 404 and is not executed.
- `GET /public/chat/<room>/` returns 403 or 404, not a directory listing.
